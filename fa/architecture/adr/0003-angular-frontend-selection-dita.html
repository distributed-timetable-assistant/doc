DiTA — دستیار جدول زمانی توزیع‌شده

0002. انتخاب Angular برای فرانت‌اند DiTA

تاریخ: 2026-06-19

وضعیت

پذیرفته‌شده

زمینه (Context)

DiTA یک محصول وب‌محور، چندنقشی و enterprise-grade است که در آن چند پورتال و چند نوع جریان کار هم‌زمان وجود دارد: کاربر، مؤسسه، سرویس‌ها، داشبوردهای مدیریتی، فرم‌های پیچیده، جدول‌های داده، گزارش‌ها و نمودارها.

در چنین محصولی، مسئله فقط «رندر کردن UI» نیست؛ مسئله این است که:

  • ساختار پروژه در طول زمان قابل پیش‌بینی بماند.
  • جداسازی مسئولیت‌ها در سطح فایل و ماژول enforce شود.
  • اضافه شدن اعضای جدید تیم هزینه‌ی زیادی ایجاد نکند.
  • توسعه‌ی ویژگی‌های UI در مقیاس بزرگ به شلختگی معماری منجر نشود.
  • اکوسیستم کتابخانه‌های enterprise برای جدول، فرم، نمودار، اعتبارسنجی و layout در دسترس و بالغ باشد.
  • عملکرد سامانه روی طیف گسترده‌ای از سخت‌افزارها، از جمله لپ‌تاپ‌های قدیمی و سیستم‌های آموزشی با منابع محدود، قابل قبول باقی بماند.

بر این اساس، انتخاب فریم‌ورک فرانت‌اند باید بر پایه‌ی نگهداشت‌پذیری بلندمدت، استانداردسازی ساختار، بلوغ اکوسیستم و عملکرد قابل پیش‌بینی انجام شود؛ نه صرفاً سرعت اولیه‌ی نمونه‌سازی.

تصمیم

Angular به‌عنوان فریم‌ورک اصلی فرانت‌اند DiTA انتخاب می‌شود.

React، Yew و Flutter برای مسیر اصلی فرانت‌اند این محصول انتخاب نمی‌شوند.

چرا React انتخاب نمی‌شود

React رویکردی کم‌نظرتر (less opinionated) نسبت به Angular دارد و بسیاری از تصمیمات معماری را به تیم توسعه واگذار می‌کند. این انعطاف‌پذیری در برخی پروژه‌ها مزیت محسوب می‌شود، اما در DiTA ترجیح داده شده است از فریم‌ورکی استفاده شود که الگوهای ساختاری مشخص‌تر و conventionهای رسمی‌تری را برای توسعه‌ی بلندمدت پروژه فراهم می‌کند.

در DiTA، ما به یک چارچوب opinionated نیاز داریم که:

  • مرزهای ماژولی را شفاف کند،
  • convention را از ابتدا تحمیل کند،
  • و اجازه ندهد UI به‌مرور به مجموعه‌ای از componentهای ناهمگن و کم‌قاعده تبدیل شود.

React این سطح از opinionation را به‌صورت پیش‌فرض فراهم نمی‌کند و برای این پروژه، ریسک drift معماری بالاتر از مزیت flexibility آن است.

چرا Yew انتخاب نمی‌شود

Yew از Rust و ماکروی html! برای تعریف view استفاده می‌کند و سینتکس آن عملاً JSX-like است. این مدل برای تیمی که به‌دنبال یک فرانت‌اند وب کاملاً enterprise و مبتنی بر اکوسیستم جاافتاده است، مزیت اصلی Angular را ندارد: بلوغ اکوسیستم UI.

در DiTA، ما به کتابخانه‌های بالغ برای این موارد نیاز داریم:

  • جدول‌های داده‌ی سنگین،
  • نمودارها و داشبوردها،
  • فرم‌های پیچیده،
  • کامپوننت‌های تعاملی سازمانی،
  • تقویم‌ها و زمان‌بندها،
  • و ابزارهای آماده‌ی طراحی و layout.

Yew از نظر فلسفه‌ی فنی جذاب است، اما برای مسیر اصلی محصول هنوز از نظر تعداد و بلوغ کتابخانه‌های UI، تجربه‌ی عملی توسعه، بازار نیروی انسانی و تنوع راهکارهای enterprise به سطح Angular نرسیده است. در نتیجه، انتخاب Yew برای DiTA ریسک پیاده‌سازی و نگهداشت را بالا می‌برد.

علاوه بر این، مدل تعریف UI در Yew مبتنی بر ماکروی html! است و جداسازی کلاسیک template، style و logic که در Angular وجود دارد را فراهم نمی‌کند. این موضوع با اهداف معماری DiTA در زمینه‌ی ساختارمند بودن کد فرانت‌اند هم‌راستا نیست.

چرا Flutter انتخاب نمی‌شود

Flutter مدل widget-centric دارد و UI را به‌صورت یک tree از widgetها تعریف می‌کند. این مدل برای اپلیکیشن‌های cross-platform بسیار خوب است، اما DiTA یک محصول وب‌محور است و نیازهای آن حول browser-native بودن، یکپارچگی با DOM، اکوسیستم وب، فرم‌ها، دسترسی‌پذیری، و الگوهای جاافتاده‌ی enterprise web می‌چرخد.

برای DiTA، Flutter چند هزینه ایجاد می‌کند:

  1. فاصله گرفتن از ecosystem اصلی وب و کتابخانه‌های بالغ آن.
  2. افزایش لایه‌ی انتزاعی در جایی که ما به کنترل دقیق بر ساختار، semantics و integration نیاز داریم.
  3. وابستگی به موتور رندر Flutter به‌جای استفاده‌ی مستقیم از DOM مرورگر.
  4. افزایش ریسک بروز مشکلات عملکردی در سناریوهای شامل نمودارها، داشبوردها، تعاملات سنگین و داده‌های حجیم.

در Flutter Web، مسیر رندر معمولاً به‌صورت زیر است:

Dart
 ↓
Flutter Engine
 ↓
CanvasKit / HTML Renderer
 ↓
Browser

در حالی که در Angular مسیر رندر به DOM مرورگر نزدیک‌تر است:

Angular
 ↓
DOM
 ↓
Browser

این تفاوت معماری به این معنا نیست که Flutter Web الزاماً کند است، اما یک لایه‌ی اضافی بین برنامه و مرورگر ایجاد می‌کند که می‌تواند در برخی سناریوها هزینه‌ی عملکردی داشته باشد.

در ارزیابی اولیه‌ی انجام‌شده روی نمونه‌ی Flutter Web، هنگام تعامل با نمودارها و عملیات hover روی سخت‌افزار میان‌رده و قدیمی، توقف‌های محسوس رابط کاربری مشاهده شد. اگرچه این نتیجه به‌تنهایی برای قضاوت قطعی درباره Flutter کافی نیست، اما نشان می‌دهد که برای محصولی مانند DiTA که متکی بر داشبوردها، نمودارها و تعاملات داده‌محور است، ریسک عملکردی بیشتری نسبت به Angular وجود دارد.

همچنین مزیت اصلی Flutter در توسعه‌ی هم‌زمان Mobile، Desktop و Web است؛ در حالی که DiTA در حال حاضر یک محصول web-first محسوب می‌شود و نیازمندی‌های اصلی آن در اکوسیستم وب بهتر پوشش داده می‌شوند.

به همین دلیل Flutter برای این محصول مناسب‌ترین گزینه‌ی اصلی نیست.

چرا Angular انتخاب می‌شود

Angular برای DiTA مناسب‌تر است چون:

  • template-first و opinionated است و ساختار را از ابتدا enforce می‌کند.
  • برای پروژه‌های سازمانی با lifecycle طولانی، convention روشن‌تری دارد.
  • جداسازی template، style و logic را به شکل طبیعی‌تری فراهم می‌کند.
  • برای تیم‌های چندنفره، code review و نگهداشت را ساده‌تر می‌کند.
  • در اکوسیستم وبِ enterprise، برای جدول، فرم، layout، نمودار و visualization مسیرهای جاافتاده‌تری دارد.
  • به‌صورت مستقیم با DOM و قابلیت‌های استاندارد مرورگر کار می‌کند.
  • برای طیف گسترده‌تری از دستگاه‌ها و سخت‌افزارها رفتار قابل پیش‌بینی‌تری دارد.
  • با نیازهای DiTA در پورتال‌های متعدد، داشبوردها، و نمودارهای داده‌محور سازگارتر است.

در یک کلام، Angular برای DiTA انتخابی است که هزینه‌ی شلختگی معماری را پایین می‌آورد، ریسک عملکردی را کاهش می‌دهد و قابلیت رشد محصول را در بلندمدت افزایش می‌دهد.

پیامدها (Consequences)

پیامدهای مثبت

  • ساختار فرانت‌اند قابل پیش‌بینی و قابل enforce شدن می‌شود.
  • کد UI در مقیاس بزرگ کمتر به هم می‌ریزد.
  • onboarding توسعه‌دهنده‌ی جدید ساده‌تر می‌شود.
  • استفاده از کتابخانه‌های بالغ وب برای نمودارها، جداول و داشبوردها ساده‌تر خواهد بود.

هزینه‌ها و ریسک‌ها

  • boilerplate اولیه از React بیشتر است.
  • یادگیری Angular برای اعضای ناآشنا زمان‌برتر است.
  • اگر discipline تیمی ضعیف باشد، حتی Angular هم می‌تواند شلوغ شود.
  • برای رسیدن به معماری خوب، نیاز به convention و review policy مشخص وجود دارد.

راهکار کاهش ریسک

  • تعریف معماری ماژولار روشن از ابتدا.
  • استفاده از convention ثابت برای feature module، component، service و model.
  • محدود کردن componentهای بزرگ و شکستن UI به واحدهای کوچک.
  • استفاده از یک design system داخلی و یک لایه‌ی UI مشترک.
  • الزام به code review برای جلوگیری از drift معماری.
  • تعریف معیارهای عملکردی (Performance Budget) و پایش مستمر آن‌ها در فرایند توسعه.

نتیجه

برای DiTA، Angular بهترین انتخاب اصلی برای فرانت‌اند است؛ چون هم‌زمان چهار نیاز کلیدی را پوشش می‌دهد: ساختار، نگهداشت‌پذیری، بلوغ اکوسیستم وب و عملکرد قابل پیش‌بینی.

React، Yew و Flutter هر کدام در زمینه‌هایی قابل دفاع‌اند، اما برای نیازهای فعلی و مسیر رشد DiTA، مزیت آن‌ها بر هزینه و ریسکشان غلبه نمی‌کند.
